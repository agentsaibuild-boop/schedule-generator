## РАЗДЕЛ А: Конвертиране на документи

**#01: PyPDF2 е неподходящ за ВиК тръжна документация**
PyPDF2 се проваля при сканирани и AutoCAD-генерирани PDF файлове.
Замяна: PyMuPDF (fitz) — 3-степенна стратегия:
GOOD (fitz директно) → PARTIAL (DeepSeek reformat) → SCANNED (vision OCR).
Тест: 31 PDF от проект Герман — 27 GOOD, 1 PARTIAL, 3 SCANNED.

**#02: AutoCAD PDF ситуации не дават четим текст с fitz**
Ситуационните чертежи (трасировъчни планове) са векторни, но текстът им
излиза като фрагментирани координати при fitz извличане.
Решение: vision OCR (Anthropic Claude) за всяка страница — чете визуалните
надписи на улиците, включително завъртяния текст по оста на улицата.

**#03: Имената на улиците в ситуацията са ЗАВЪРТЕНИ по оста на улицата**
При vision OCR на ситуации — инструктирай AI да търси завъртян/наклонен текст,
не само хоризонтален. На кръстопати има ДВЕ перпендикулярни улици.
Стрелките „посока на отвеждане" съдържат „към ул. X" — извличай само X.

## РАЗДЕЛ Б: AI генериране — халюцинации и качество

**#04: AI ИЗМИСЛЯ имена на улици (халюцинации по места)**
При генериране на задачи AI добавя измислени имена на улици, квартали
и местности, които не присъстват в документацията.
Решение: whitelist от реални топоними → подаден в промпта като „ДОПУСТИМИ ИМЕНА".
Валидация след генериране с `_validate_task_locations()`.

**#05: AI анализира само имената на файловете, не съдържанието им**
Ранна грешка: `analyze_documents()` подаваше само списък с имена на файлове.
AI генерираше графика „по заглавие", не по реалните количества от КСС.
Решение: `get_all_text()` → пълното съдържание се подава в промпта (до 120k знака).

**#06: Без КСС файл AI генерира измислен график**
Ако в папката няма КСС (количествено-стойностна сметка), AI генерира
произволен смислово изглеждащ график без реална основа.
Решение: `classify_files()` блокира генерирането при липса на КСС.

**#07: Правило #80 е абсолютно — само дейности от КСС**
AI не трябва да добавя дейности, невидими в КСС, дори ако са „логични".
Изключения: мобилизация/демобилизация (стандартни начало/край).

**#08: Keyword matching за намерение не работи при свободен текст на български**
Ранна имплементация: `if "генерирай" in message or "график" in message`.
Пропуска: „направи ми разпис", „искам grafik", „пусни го".
Решение: DeepSeek intent router с пълен prompt — разбира свободна формулировка.

## РАЗДЕЛ В: Типове проекти — специфики

**#09: Разпределителна мрежа — дезинфекция е PER SECTION, не накрая**
При разпределителна мрежа с много участъци дезинфекцията се прави
за всеки участък поотделно, след хидравличната проба на същия участък.
НЕ се прави обща дезинфекция накрая.

**#10: Довеждащ водопровод — дезинфекция е СЛЕД ВСИЧКИ секции**
Обратното на #09: при довеждащ водопровод (1 нишка) дезинфекцията
е единична задача след полагането на целия тръбопровод.

**#11: Rolling Wave — Вода → Канал → Пътни с 10-12 дни LAG**
Стандартният ред при паралелни мрежи. Пътните работи НЕ приключват
преди канализацията — финалният асфалт чака цялата канализация.

**#12: ИНЖЕНЕРИНГ — Авторски надзор е непрекъсната дейност**
Авторският надзор = продължителност на Фаза 2 (Строителство).
Стартира с Протокол обр.2 (Откриване на строителна площадка).
Приключва с Констативен акт обр.15 (Годност за приемане).

**#13: ИНЖЕНЕРИНГ — Продължителност 780-1000 дни е типична**
Фаза 1 Проектиране: 120-180д
Фаза 2 Строителство: 400-600д
Фаза 3 Авторски надзор: = продължителността на Фаза 2

**#14: Pipeline overlap при разпределителна мрежа = 16 дни**
Следващият участък стартира 16 дни след предходния (не след приключването му).
Позволява 2-3 паралелни екипа да работят едновременно.

**#15: SS+1d overlap за довеждащ водопровод**
combined_days = max(exc_days, pipe_days + 1)
Изкопът и полагането на тръбата вървят паралелно с 1 ден изпреварване.

## РАЗДЕЛ Г: Производителности

**#16: Ефективна производителност ≠ техническа производителност**
Ефективната производителност е ограничена от НАЙ-БАВНАТА операция в цикъла.
DN90 PE: изкоп 57.6 м/д, полагане 144 м/д → ефективна 12 м/д (ограничено от изкопа).

**#17: Терен фактори са задължителни**
Асфалт: ×0.75 (разрязване + възстановяване)
Черен път: ×1.0 (базово)
Горски/скален: ×0.6 (трудни условия)
Без коректор — продължителностите са занижени.

**#18: Производителностите v0.4 са верифицирани — не ги редактирай без проект**
`config/productivities.json` е тестван срещу реални проекти.
Промяна без обосновка нарушава всички параметрични изчисления.

## РАЗДЕЛ Д: Формати и експорт

**#19: XML за MS Project ЗАДЪЛЖИТЕЛНО с DurationFormat=5 и Manual=1**
DurationFormat=5 = дни (не часове или минути).
Manual=1 = ръчно планиране (задължително за MSPDI импорт).
Без тях MS Project интерпретира неправилно.

**#20: Summary tasks в MS Project НЕ получават ресурс/екип**
WBS йерархия: summary tasks са само контейнери.
Ресурсите/екипите се задават само на листови (leaf) задачи.

**#21: Кирилица в PDF изисква DejaVu Sans шрифт**
Стандартните PDF шрифтове (Helvetica, Times) не поддържат кирилица.
Задължително: `fonts/DejaVuSans.ttf` — вграден в проекта.

**#22: JSON → XML → .mpp е крайният pipeline за клиента**
Клиентът получава .mpp (MS Project native).
AI генерира JSON → export_xml.py → MSPDI XML → MS Project → Save As .mpp.

## РАЗДЕЛ Е: Системни грешки (открити при разработката)

**#23: Schedule съхранен като JSON string причинява AttributeError**
Ако `schedule` се запише в session_state като string (serialized JSON)
вместо list, всяко `.get()` или итерация дава AttributeError.
Решение: `_ensure_schedule_list()` хелпър на 4 места в app.py.

**#24: Corrupt last_schedule замества demo данните**
Ако `projects_history.json` съдържа невалиден JSON в `last_schedule`,
приложението презаписва demo графика с None/празно.
Решение: `if parsed:` проверка преди restore.

**#25: Дублиране на чат история — 3 независими бъга**
1. Double restore в auto-restore блока
2. `_save_chat_history()` конкатенираше вместо да замества
3. Welcome съобщение се добавяше отново при всеки reload
Решение: `welcome_shown = True` при auto-restore, преписана save функция.

**#26: PowerShell folder dialog изисква -STA флаг**
`[System.Windows.Forms.FolderBrowserDialog]` изисква Single-Thread Apartment.
Без `-STA`: dialog не се отваря или хвърля COM грешка.
Команда: `powershell -STA -Command "..."`.

**#27: Pre-commit hook НЕ трябва да пуска E2E тестове автоматично**
E2E тестовете изискват: стартирано приложение + реални API ключове + интернет.
На CI/CD машина или при commit без стартиран app — hook се проваля.
Решение: pre-commit пуска само unit тестове; E2E — ръчно.

**#28: Case-sensitive проверка в _SKIP_WORDS дава фалшиви hallucination warnings**
Regex връща „Монтаж" (главна буква), но set съдържа само „монтаж" (малка).
`if token in _SKIP_WORDS` → никога не match-ва → всяка стандартна дума изглежда като халюцинация.
Решение: `if token.lower() in _SKIP_WORDS`.

## РАЗДЕЛ Ж: Архитектурни решения

**#29: 3-нивова knowledge система (lessons → methodologies → skills)**
Нивото се избира спрямо операцията:
- minimal: само rules (за бързи чат отговори)
- full: rules + methodologies (за генериране)
- verification: всичко (за проверка на генериран график)

**#30: DeepSeek = работник, Anthropic = контрольор**
DeepSeek V3 генерира (евтино, бързо).
Anthropic Sonnet верифицира (скъпо, точно).
Fallback: ако DeepSeek не е наличен → Anthropic генерира директно.

**#31: Моделът (референтен проект) печели при конфликт с уроци**
Ако имаш референтен проект (Плевен, Харманли) — копирай неговата структура.
Общите правила са по-слаби от конкретния модел. (Правило #70)
