# MS Project структура — Експертно ниво

## КРИТИЧНО: Пълният pipeline на изходния файл

```
AI генерира JSON
      ↓
app.py → export_xml.py → MSPDI XML (.xml)
      ↓
Потребителят отваря XML в MS Project:
  File → Open → XML Format
      ↓
MS Project показва графика
      ↓
File → Save As → Project Files (.mpp)
      ↓
Крайният .mpp файл (нативен MS Project формат)
```

**Нямаме библиотека за директно генериране на .mpp — минаваме задължително през XML.**
Качеството на .mpp зависи изцяло от качеството на JSON-а, генериран от AI.

## WBS йерархия (Work Breakdown Structure)

MS Project работи с дърво от задачи. Ниводете (OutlineLevel) са:

```
OutlineLevel 0  → Проект (UID=0, root task, задължителен)
OutlineLevel 1  → Фаза / Раздел (Summary Task — обобщаваща)
OutlineLevel 2  → Участък / Улица (Summary Task или листова задача)
OutlineLevel 3  → Конкретна дейност (листова задача, с ресурс и продължителност)
```

### Примерна WBS за ВиК проект:

```
0. Герман Водопровод (root, UID=0)
  1. ФАЗА 1 — ПРОЕКТИРАНЕ (Summary, Level 1)
    1.1 Идеен проект (Level 2, листова)
    1.2 Технически проект (Level 2, листова)
  2. ФАЗА 2 — СТРОИТЕЛСТВО (Summary, Level 1)
    2.1 ул. Петрова (Summary, Level 2) ← участък
      2.1.1 Земни работи (Level 3, листова, DN=160, L=120м)
      2.1.2 Монтаж тръби (Level 3, листова)
      2.1.3 Засипване (Level 3, листова)
    2.2 ул. Иванова (Summary, Level 2)
      2.2.1 Земни работи (Level 3, листова)
      ...
  3. ФАЗА 3 — АВТОРСКИ НАДЗОР (Summary, Level 1)
    3.1 Авт. надзор водопровод (Level 2, листова)
```

### Правила за Summary Tasks:
- Summary task = задача с деца (под-задачи)
- Summary tasks **НЕ** получават ресурс (екип)
- Продължителността им се **изчислява автоматично** от MS Project като span от първото до последното дете
- В JSON: `"sub_activities": [...]` → parent се маркира като summary

## Типове задачи в MS Project

| Тип | Summary | Milestone | Duration | Ресурс |
|-----|---------|-----------|----------|--------|
| Summary Task | 1 | 0 | auto | НЕ |
| Нормална задача | 0 | 0 | >0 дни | ДА |
| Milestone (етап) | 0 | 1 | 0 | НЕ |

### Кога се използва Milestone:
- Завършване на фаза: "Проектирането завършено"
- Подаване на документация: "Подадено разрешение за строеж"
- Критична точка в строителството

## Зависимости (Predecessor Links)

MS Project поддържа 4 типа:

| Тип | Код | Значение | Кога |
|-----|-----|----------|------|
| FS | 1 | Finish-to-Start | Стандартно: Б започва след А |
| SS | 3 | Start-to-Start | Паралелна работа: Б започва когато А започне |
| FF | 2 | Finish-to-Finish | Б завършва когато А завършва |
| SF | 4 | Start-to-Finish | Рядко |

### LAG (изчакване между задачите):
- `lag > 0` = изчакване в дни (напр. 3 дни за съхнене на бетон)
- `lag < 0` = припокриване (напр. засипването започва 2 дни преди монтажа да е завършен)

### Типични зависимости за ВиК:
```
Земни работи  →[FS+0]→ Монтаж тръби
Монтаж тръби  →[FS+1]→ Изпитване под налягане  (1 ден изчакване)
Изпитване     →[FS+0]→ Засипване
Водопровод ул.А →[FS+10]→ Канализация ул.А  (Rolling Wave, 10д LAG)
Канализация   →[FS+0]→ Пътни работи
```

## Manual vs Auto scheduling

**Нашият XML използва Manual=0 + ConstraintType=2 — ПРАВИЛНО.**

- `Manual=1`: ГРЕШНО — слага пин икона (📌) в колона "Task Mode" на всяка задача. Изглежда непрофесионално.
- `Manual=0` (Auto) + `ConstraintType=2` (Must Start On) + `ConstraintDate`: ПРАВИЛНО — датите са заключени без пин икони.

За строителни графици: `Manual=0`, `ConstraintType=2`, `ConstraintDate=start_date`. Така датите се спазват точно и файлът изглежда като професионално ръчно създаден MS Project план.

## Какво AI трябва да знае при генериране на JSON

### 1. Всяка дейност от КСС = отделна задача в MS Project
```json
{
  "id": "T1",
  "name": "Земни работи — ул. Петрова DN160",
  "start_day": 1,
  "duration": 15,
  "end_day": 15,
  "team": "Екип А",
  "diameter": "DN160",
  "length_m": 120,
  "type": "water_supply",
  "phase": "construction",
  "dependencies": []
}
```

### 2. Правилна WBS йерархия в JSON
```json
{
  "id": "PHASE2",
  "name": "СТРОИТЕЛСТВО",
  "type": "summary",
  "phase": "construction",
  "start_day": 181,
  "duration": 0,
  "sub_activities": [
    {
      "id": "SEC1",
      "name": "ул. Петрова",
      "type": "summary",
      "sub_activities": [
        {"id": "T1", "name": "Земни работи", ...},
        {"id": "T2", "name": "Монтаж тръби", ...}
      ]
    }
  ]
}
```

### 3. Задължителни полета за добър .mpp
- `id` — уникален идентификатор (напр. "T1", "T2", ...)
- `name` — пълно наименование от КСС
- `start_day` — ден на старт (1 = първи ден на проекта)
- `duration` — продължителност в дни
- `end_day` — last day (start_day + duration - 1)
- `dependencies` — списък от id-та на предшественици
- `team` — екип/бригада (= ресурс в MS Project)
- `type` — тип дейност (water_supply, sewage, road_works, design, supervision...)
- `phase` — фаза (design, construction, supervision)
- `is_critical` — критичен ли е (bool)

### 4. Полета специфични за ВиК (custom fields в .mpp)
- `diameter` — диаметър на тръбата (напр. "DN160", "DN250")
- `length_m` — дължина в метри (число)
- `unit` — мярка (напр. "м", "бр.", "м²")

## Критични грешки, които разрушават .mpp

1. **Липсващ UID=0 root task** — MS Project не може да отвори файла
2. **DurationFormat липсва или е грешен** — дните се показват като минути
3. **Summary task с ресурс** — MS Project показва грешно натоварване
4. **duration=0 без Milestone=1** — задачата не се вижда на Gantt
5. **Наследени зависимости към несъществуващ UID** — MS Project игнорира тихо или крашва
6. **Отрицателни start_day** — датите излизат преди начало на проекта
7. **end_day < start_day** — невалидна задача

## Ръчно vs AI-генерирано .mpp — ключови разлики

### При ръчно създаване в MS Project:
- Потребителят задава WBS структурата сам
- Summary tasks групират логически участъците
- Зависимостите отразяват реалния строителен процес
- Ресурсите (екипи) са назначени само на листови задачи
- Именуването следва КСС точно (номер + наименование)

### AI трябва да имитира това:
- Да НЕ генерира плосък (flat) списък от задачи
- Да създава Summary tasks за всяка фаза и участък
- Да именува задачите точно по КСС (с номер ако има)
- Да назначава екипи само на листови задачи (не на summary)
- Да добавя Milestone задачи за важни етапи

## Правило #85: JSON → XML → .mpp pipeline е задължително знание

Крайният продукт за клиента е **.mpp файл**. AI генерира JSON, Python го конвертира в MSPDI XML, потребителят го отваря в MS Project и записва като .mpp. Всичко в JSON трябва да е структурирано така, че да изглежда като ръчно създаден MS Project план — с WBS йерархия, правилни зависимости, summary tasks и листови задачи с ресурси.
